# CLAUDE.md — EXPLAIN_A

## 프로젝트
학생이 음성으로 개념을 **설명하면** AI가 듣고 **소크라테스식 질문**을 던지는 역방향 학습 앱(학생 → AI).
설명 데이터를 **LEI(Learning Explanation Index, 0~100)** 로 수치화한다.

- **스택**: HTML+CSS+JS 단일 파일 · Supabase(DB/Auth) · Gemini(STT + 질문 생성 + 채점) · GitHub Pages
- **핵심 파일**:
  - `index.html` — 앱 전체(5화면: 로그인/홈/학습/결과/교사 대시보드 + 연동 로직)
  - `schema.sql` — Supabase 테이블 + RLS + 가입 트리거
  - `README.md` — 설정·배포·보안 가이드
- **LEI = 5개 지표 평균**: 개념포함도 · 설명정확도 · 응답속도 · 설명확장성 · 질문대응력

## 프로젝트 규칙
- **단일 파일 유지**: 앱 로직은 `index.html` 한 파일에. 별도 빌드 도구·프레임워크 도입 금지(요청 없으면).
- **데모 모드 우선**: 키가 없어도 항상 동작해야 한다. 실연동 코드 추가 시 데모 폴백(`mock*` 함수)을 깨지 말 것.
- **주석은 한국어**, 기존 코드 스타일·들여쓰기·네이밍을 따른다.
- **키 노출 주의**: GitHub Pages는 정적 호스팅. 키는 localStorage에만 저장. 운영 시 Gemini 호출은 Edge Function 프록시로.
- **리스크 대응 기능 보존**: 이의 제기(오탐 완충), 연속학습/격려(동기부여), KPI 분리 — 리스크분석 권고 반영분이니 임의 제거 금지.

## 작업 원칙 (Karpathy Guidelines)
LLM 코딩의 흔한 실수를 줄이기 위한 행동 지침. 코드를 쓰거나 리뷰·리팩터링할 때 적용한다.

### 1. 코딩 전에 생각하라 — "추측하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러내라"
- 가정을 명확히 말하고, 불확실하면 질문한다.
- 해석이 여러 개면 임의로 정하지 말고 선택지를 제시한다.
- 더 직접적인 해법이 있으면 그것을 권한다.
- 막히는 지점을 멈춰서 짚고 도움을 요청한다.

### 2. 단순함이 먼저다 — "문제를 푸는 최소 코드. 추측성 코드 금지"
- 요청하지 않은 기능은 넣지 않는다.
- 한 번만 쓰는 곳에 추상화를 만들지 않는다.
- 선제적 설정 가능성(옵션화)을 추가하지 않는다.
- 현실에서 일어날 수 없는 엣지 케이스를 처리하지 않는다.
- 장황한 해법은 크게 줄인다.
- 자문: *숙련된 엔지니어가 이걸 보면 불필요하게 복잡하다고 할까?*

### 3. 외과수술적 변경 — "꼭 필요한 것만 건드리고, 네가 만든 것만 치운다"
- 근처 코드·문서·공백을 덤으로 개선하지 않는다.
- 잘 동작하는 부분은 손대지 않는다.
- 기존 컨벤션을 유지한다.
- 고아 코드는 말없이 지우지 말고 표시(플래그)한다.
- 단, 내 수정으로 새로 불필요해진 import·변수·함수는 정리한다. 기존부터 있던 미사용 코드는 명시 요청 없으면 보존한다.
- 검증: 모든 변경은 요구사항과 직접 연결되어야 한다.

### 4. 목표 주도 실행 — "성공 기준을 정의하고, 검증될 때까지 반복하라"
- 요구사항을 측정 가능한 목표로 바꾼다:
  - "검증 추가" → 엣지 케이스 테스트가 통과
  - "버그 수정" → 테스트로 재현 후 해결
  - "X 리팩터링" → 전 과정에서 테스트 통과 유지
- 다단계 작업은 단계 전/중/후에 검증 체크포인트를 둔다.
- 측정 가능한 기준은 독립적 반복을 가능케 한다. 모호한 목표는 계속 되묻게 만든다.
